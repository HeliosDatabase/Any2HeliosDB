# HeliosDB-Lite — fix regression benchmark (2026-06-23)

A/B gate for the two **applied** Lite fixes (planner `object_name_to_string`
quoted-identifier handling + `check_table_privilege` owner-key normalization),
which sit on the query-planner and per-statement privilege hot paths.

## Method
- **Baseline** = current Lite tree on `improvements-stubs-2026-06-10` with **only
  my 3 fix files reverted** (`git stash push -- src/sql/evaluator.rs src/lib.rs
  src/sql/planner.rs`), release build → `/tmp/hl-baseline`.
- **Fixed** = same tree with the fixes → `/tmp/hl-fixed`.
- Both run `start --memory` on isolated ports; identical workload.
- **Metric = point-SELECT throughput** (`SELECT v FROM b WHERE id = <lit>`),
  median of 3×5000 ops. SELECT is the path most sensitive to the planner +
  privilege changes, and is fast; the single-row *write* path is ~300 ms/row on
  this WIP branch (a separate storage concern, unrelated to these fixes), so it
  is excluded from the gate.
- Tooling: `tools/bench_oltp.py` (SELECT-dominant, literal SQL) / direct A/B.

## Result

| Build | SELECT throughput | ratio vs baseline |
|---|---|---|
| baseline | 7,139 sel/s | 1.000× |
| fixed | 7,066 sel/s | **0.990×** |

**Verdict: no regression** (0.990× ≥ 0.97 threshold; within run-to-run noise).

Note: an initial `run_ab_bench` pass mis-measured (~100 sel/s) due to CPU
contention from leftover diagnostic servers; clearing them restored the true
~7k sel/s. Always clear stray bench servers (`pgrep -x hl-baseline | xargs kill`)
before a run.
