# Any2HeliosDB v2.0.0 roadmap — AI-native procedural migration

v1.0.0 migrates the **data tier** completely (tables, columns, types, PK/FK/CHECK,
indexes, sequences, views) and **surfaces** procedural / advanced objects — stored
routines (PROCEDURE / FUNCTION / PACKAGE), triggers, materialized views, and
partitioning — for manual porting (counted in `a2h assess`, gap-reported, and
written verbatim to a `<schema>.review.sql` companion). See
[../reference/oracle-object-support.md](../reference/oracle-object-support.md).

v2.0.0 closes that gap the **AI-native** way: connect a2h to a language model and
let it **adapt each object to the target dialect, test it in an ephemeral sandbox,
and migrate working code — automatically**.

---

## The approach

a2h connects to a **pluggable LLM backend** — a local **Ollama** model or any
**RESTful / OpenAI-compatible** endpoint, so the operator chooses privacy/local
vs. hosted/capability. For each procedural object, processed in **dependency
order**:

1. **Analyze — object + its dependencies.** a2h assembles a per-object context:
   the object's verbatim source (already captured by v1.0.0 introspection) plus
   everything it depends on — called routines, referenced tables / views / types /
   sequences, package members — and the target's capability surface (what
   HeliosDB / PostgreSQL supports). A callee is migrated before its caller.
2. **Adapt.** The model rewrites the object for the target database using that
   dependency context plus its own knowledge of Oracle PL/SQL and
   PostgreSQL / HeliosDB PL/pgSQL.
3. **Test — ephemeral, metadata-only sandbox.** a2h applies the candidate to a
   throwaway **sandbox copy of the schema (metadata only — the migrated DDL, no
   production data)**: it must **create / compile**, and a smoke test must run
   (call the function, fire the trigger, refresh the mview). Failures are fed back
   to the model and it **iterates** until the object works — or a bounded retry
   budget is reached.
4. **Migrate.** The working, target-adapted object is applied to the real target,
   tagged with provenance (model, attempts, tests passed).

The result: procedural code migrated **quickly, as working target-native code** —
generalized by the model, not hand-mapped construct-by-construct.

> ### Success factor (disclaimer)
> The quality of this migration is **bounded by the LLM's knowledge** of the
> source (Oracle PL/SQL) and target (PostgreSQL / HeliosDB PL/pgSQL) dialects. A
> more capable model adapts more objects, more correctly; a smaller/local model
> adapts fewer. **The chosen model is the single biggest determinant of success.**
> a2h provides the analyze → adapt → test → iterate harness and the safety gate —
> the *translation intelligence is the model's*.

## Why not a deterministic transpiler (obsolete in the AI era)

The classic approach — a hand-written, rule-based PL/SQL → PL/pgSQL transpiler
(parser + AST + a bespoke codegen rule per construct, à la Ora2Pg) — is
**deliberately not pursued**. It is a ~20-year, permanently-incomplete effort:
every Oracle construct, builtin (`DBMS_*` / `UTL_*`), and idiom needs its own
rule, and the long tail never closes. In the AI era a model with per-object
dependency context + a sandbox test-loop **subsumes** that work — it generalizes
across constructs *without* a rule per construct, and the test gate keeps it
honest. The deterministic-transpiler track is therefore treated as **legacy /
obsolete**; a2h goes AI-native.

## Safety & guardrails

- **Nothing is accepted unless it creates and passes its smoke test** in the
  ephemeral sandbox — the model never ships unverified code.
- The sandbox is **metadata-only and ephemeral** (a throwaway schema copy), so
  testing is fast and never touches production data.
- **Opt-in** — a model backend must be explicitly configured. **Provenance** is
  recorded per object (model, attempt count, tests passed).
- **Fallback to the estimation review.** An object the model can't get working
  within the retry budget is **not shipped broken** — it falls back to v1.0.0's
  surface plus a pinpointed **Refactoring Estimation Review** (the exact blocker,
  complexity, effort estimate), so the residual manual work is a precise list.
- **Fix-in-HeliosDB.** When the blocker is a missing target builtin, the review
  emits a target-gap so the capability converges in HeliosDB (the governing
  principle) instead of the tool faking it.

## Proposed components

| Component | Purpose |
|---|---|
| `llm/backend.py` | pluggable model client — **Ollama** (local) + **RESTful / OpenAI-compatible** endpoints; configured in `config.toml` |
| `plsql/deps.py` | per-object **dependency analysis** + dependency-ordered processing |
| `plsql/adapt.py` | the **analyze → adapt → test → iterate** loop |
| `sandbox/ephemeral.py` | throwaway **metadata-only** schema copy for compile + smoke testing |
| `assess/estimation.py` | the Markdown **Refactoring Estimation Review** for objects that can't be auto-migrated |
| CLI | `a2h migrate --ai-rewrite` (adapt + test + migrate); `a2h assess --estimate-refactor` (estimate without applying) |
