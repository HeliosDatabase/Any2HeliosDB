# Any2HeliosDB v2.0.0 roadmap — procedural-object migration

v1.0.0 migrates the **data tier** completely (tables, columns, types, PK/FK/CHECK,
indexes, sequences, views) and **surfaces** procedural / advanced objects — stored
routines (PROCEDURE / FUNCTION / PACKAGE), triggers, materialized views, and
partitioning — for manual porting (counted in `a2h assess`, gap-reported, and
written verbatim to a `<schema>.review.sql` companion). See
[../reference/oracle-object-support.md](../reference/oracle-object-support.md).

v2.0.0 turns that *surface* into **automated migration where it's safely possible,
and a precise effort estimate where it isn't**. Two tracks.

---

## Track A — PL/SQL → PL/pgSQL auto-translation

Mechanically translate the objects v1.0.0 surfaces for review:

- **Procedures / functions** → PL/pgSQL function bodies (control flow, `%TYPE`,
  cursors, exceptions, `RETURN`), with anything genuinely non-portable routed to
  Track B.
- **Packages** → a schema (namespace) of PL/pgSQL functions; package state →
  session-local GUCs / temp structures.
- **Triggers** → a PL/pgSQL trigger function + `CREATE TRIGGER`.
- **Materialized views / partitioning** → native target DDL when the edition
  supports it; otherwise Track B.

Deterministic, rule-based translation handles the well-understood constructs
(`NVL`/`DECODE`/`SYSDATE`/…, simple control flow). It is correct-by-construction
but only covers what the rules know.

---

## Track B — AI-assisted live-rewrite + effort estimation (new)

For everything Track A can't translate mechanically, run a **try-rewrite →
test-on-the-fly → accept-or-estimate** pipeline, per object. v1.0.0 already
captured each object's verbatim source, so this builds on that — it doesn't
re-read the source.

### Pipeline (per object)

1. **Attempt a rewrite.** First the Track-A deterministic translator; for the
   remainder, an **AI-assisted rewrite** (the configured model) proposes a
   PL/pgSQL equivalent of the object's source.
2. **Test it on-the-fly.** Apply the candidate to the live target inside a
   savepoint / throwaway schema, then verify:
   - it **creates/compiles** on the target;
   - a **smoke test** runs — call the function with sampled arguments / fire the
     trigger on a test row / `SELECT` from the materialized view;
   - where source data is reachable, a **differential test** compares the
     rewrite's output against the Oracle original for the same inputs.
3. **Accept or fall through.**
   - **PASS** → migrate the object automatically, tagged with provenance
     (`deterministic-translated` or `ai-rewritten+tested`).
   - **FAIL / IMPOSSIBLE** → the rewrite can't be expressed or doesn't behave
     equivalently (a referenced Oracle builtin/package has **no target
     equivalent**, an **unsupported construct**, a **semantic gap**, or the
     differential test **diverges**). **Never ship a guessed/wrong object** —
     instead capture *pinpointed blockers* for that object and route it to the
     estimation review.
4. **Estimation review (Markdown).** For every object that couldn't be
   auto-migrated, emit a detailed **Refactoring Estimation Review**:
   - per object: the **exact blocker(s)** — which missing function / unsupported
     construct, **at which line** of the source;
   - a **complexity score** and an **effort estimate** (person-hours/days);
   - a **suggested manual approach** (target equivalent, rewrite pattern, or
     "fix in HeliosDB" target-gap if a builtin should be added);
   - an aggregated **refactoring-effort summary** across the schema.

### Why this shape

- **Maximizes automation** (translate + test whatever is feasible) **without ever
  silently shipping a wrong object** — the on-the-fly test gate + fall-through are
  the safety floor.
- Where automation isn't possible, the user gets a **precise, actionable effort
  breakdown** — not just "review these N objects," but *which builtin is missing,
  where, and how hard to refactor*.
- Reuses v1.0.0 building blocks: introspection (verbatim source), `assess`/gap
  (cost heuristic), the **capability probe** (what the target supports), and the
  **validation harness** (the on-the-fly test).

### Proposed components

| Component | Purpose |
|---|---|
| `plsql/translate/` | deterministic + AI-assisted rewrite engines (pluggable; AI opt-in, via the configured model) |
| `plsql/verify.py` | on-the-fly test loop — savepoint apply + smoke/differential test against the capability probe + a scratch schema |
| `assess/estimation.py` | the Markdown "Refactoring Estimation Review" generator (per-object blockers + effort) |
| CLI | `a2h migrate --rewrite[=ai]` (attempt + test rewrites during migrate); `a2h assess --estimate-refactor` (emit the estimation MD without applying) |

### Guardrails

- AI rewrites are **off by default** (explicit opt-in flag). Every AI-rewritten
  object **must pass the on-the-fly test** before it is accepted.
- No object is auto-migrated unless its test passes; everything else falls back
  to the v1.0.0 behavior (review file + estimation report) — the safe floor never
  regresses.
- **Provenance** is recorded per object (`deterministic-translated` /
  `ai-rewritten+tested` / `manual-required`) so a migration is fully auditable.
- Consistent with the governing principle: when a rewrite is blocked by a missing
  target builtin, the estimation review emits a **target-gap** pointing at the
  HeliosDB fix, so capability gaps converge in HeliosDB rather than thickening the
  tool.
